# AI 数据分析平台 — 图片数据集嵌入可视化分析详细文档

## 项目总览

这是一个**计算机视觉数据集可视化分析平台**，全栈架构：**Next.js 16 + React 19（TypeScript）前端 + Python FastAPI 后端**。核心功能：用户上传 YOLO/Pascal VOC 格式的 ZIP 数据集，平台自动解析标注、提取 BGE-VL-Large 图像语义嵌入（768维）、生成 PCA 二维散点可视化，提供基于高维嵌入的**离群检测与近重复识别**清洗能力，以及基于多模态大模型的**零样本语义属性自动标注**。

整体分为两大模块：
1. **嵌入可视化分析模块** — ZIP 解析 → BGE-VL-Large 嵌入提取 → 分块并行加速 → PCA 降维可视化
2. **数据清洗与语义标注模块** — kNN 离群检测、互最近邻重复识别、零样本语义分类

---

## 一、嵌入可视化分析模块

### 1.1 整体处理流水线

用户上传 ZIP 数据集后，后端按两阶段处理：

```
阶段一（快速预览）：
ZIP 上传 → SHA-256 去重检查 → 移动归档 → 解压
→ 自动发现数据集根目录 → 并行解析图片+标注（Pascal VOC / YOLO）
→ 生成占位 2D 坐标 → 写入 Milvus → 返回基础 payload → 前端展示图片网格

阶段二（后台补全）：
加载图片绝对路径 → BGE-VL-Large 图像嵌入推理
→ 嵌入缓存写入（Milvus，SHA-256 键） → PCA 降维到 2D
→ 语义属性零样本分类 → 离群/重复清洗预计算
→ 更新 payload → 前端刷新散点图和清洗叠加层
```

**关键设计：两阶段返回策略。**
- 第一阶段：只做文件解析，前端秒出图片网格和标注框，不等待模型推理。
- 第二阶段：后台异步跑 BGE embedding + PCA + 语义分类 + 清洗，完成后前端自动刷新。
- 两个阶段由 `create_dataset_without_embedding()` 和 `complete_dataset_embedding()` 分别完成，
  中间通过线程 job 队列管理，前端轮询 `/api/dataset/jobs/{job_id}` 获知进度。

### 1.2 ZIP 上传与解压

**入口函数：** `DatasetService.start_upload_job()` → `_run_upload_job()`

1. 前端通过流式上传 ZIP 到 `/api/dataset/upload`，后端边接收边计算 SHA-256。
2. 用内容哈希查 `_cache/{hash}/`，命中则直接返回缓存（跳过全部处理）。
3. 未命中：ZIP 暂存到 `_incoming/{job_id}/`，启动后台线程执行 `_run_upload_job`。
4. `create_dataset_without_embedding()` 中将 ZIP 从 `_incoming/` **移动到**
   `.dataset-store/{dataset_id}/archive/`（不拷贝，省一次完整 I/O），然后解压到 `extracted/`。
5. `find_dataset_root()` 自动发现数据根目录：检查当前目录或一级子目录是否包含
   `images/` 或 `train/val/test` 分割目录。

**安全措施：** `_safe_extract_zip()` 检查每个 ZIP 条目的路径，防止路径穿越攻击。

### 1.3 标注解析（并行化）

**入口函数：** `DatasetService.analyze_dataset_directory()`

支持两种标注格式，解析逻辑抽成纯函数 `_parse_single_image()`，用 `ThreadPoolExecutor`
并行处理（I/O 密集型：读图片尺寸、XML/TXT 文件）：

- **Pascal VOC XML**：解析 `<object>` → `<name>`（类别）+ `<bndbox>`（像素坐标），
  归一化为 `[x, y, w, h]`（0~1）。
- **YOLO TXT**：每行 `class_id cx cy w h [confidence]`，已是归一化坐标。类别名用 `class_{id}`。
- 结果按 split + filename 排序恢复确定性顺序，保证每次上传同一批数据输出一致。
- 2D 坐标先填占位值 `[0.0, 0.0]`，等第二阶段 BGE+PCA 完成后替换。

### 1.4 BGE-VL-Large 图像嵌入提取

**模型：** BGE-VL-Large，**BAAI** 出品的图像-文本联合嵌入模型，输出 **768 维**向量。
图像和文本映射到同一语义空间，余弦相似度可直接衡量图文/图图语义距离。

#### 推理模式 A：Infinity Embedding Engine（默认）

```python
from infinity_emb import AsyncEmbeddingEngine, EngineArgs

engine = AsyncEmbeddingEngine.from_args(EngineArgs(
    model_name_or_path=str(BGE_MODEL_PATH),
    engine="torch",
    device="cuda",          # 自动检测 CUDA/CPU
    bettertransformer=True, # 启用 BetterTransformer 加速
    batch_size=64,          # BGE_BATCH_SIZE，可在环境变量调整
))
```

**引擎池设计：**
- 创建 `BGE_INFINITY_CONCURRENCY`（默认 3）个独立引擎实例。
- 每个引擎拥有独立的 asyncio 事件循环，运行在守护线程中。
- 线程安全提交：`asyncio.run_coroutine_threadsafe(coro, loop)` 将编码任务提交到指定引擎。
- 通过 `ThreadPoolExecutor` + `as_completed` 实现多 chunk 并行推理。

**为什么选择 Infinity 而非直接使用 sentence-transformers？**
- Infinity 引擎内置异步批处理队列和 BetterTransformer 优化，GPU 利用率更高。
- 多引擎池可以真正并行处理多个分块，实现 **~2.8 倍推理加速**。

#### 推理模式 B：Sentence-Transformers（降级方案）

```python
from sentence_transformers import SentenceTransformer
model = SentenceTransformer(model_path, trust_remote_code=True)
encoded = model.encode(image_paths, batch_size=64, show_progress_bar=False)
```

- 当 Infinity 引擎初始化失败或 `BGE_USE_INFINITY=0` 时使用。
- 兼容性修复 `configure_bge_sentence_transformer()`：新版 sentence-transformers 对
  BGE-VL CLIP 模块会尝试索引 `pooler_output` 键，但模型直接返回嵌入张量，导致
  `too many indices for tensor of dimension 2`。修复方式为将 `method_output_name` 从
  `"pooler_output"` 改为 `None`。

### 1.5 分块并行处理与嵌入缓存

**核心函数：** `get_cached_or_encode_image_embeddings()`

这是实现 **2.8 倍加速**的关键所在。处理分为四步：

**Step 1：逐图查缓存**
```python
for image in images:
    digest = sha256(file_content)  # 文件内容哈希，非文件名
    cached = milvus_cache.get(digest)
    if cached: vectors.append(cached)   # 命中
    else:      missing.append(image)    # 需要推理
```
- 缓存键是文件内容的 SHA-256，同一张图无论出现在哪个数据集、什么路径，只推理一次。
- 缓存在 Milvus Lite 的 `embedding_cache` 集合中，跨数据集共享。

**Step 2：分块切分**
```python
chunk_size = BGE_EMBEDDING_CHUNK_SIZE  # 默认 256
chunks = ceil(len(missing_images) / chunk_size)
```
- 分块避免一次性加载过多图片导致 GPU OOM。

**Step 3：并行编码（多引擎 + 线程池）**
```python
# concurrency = BGE_INFINITY_CONCURRENCY（默认 3）
if concurrency > 1 and len(chunks) > 1:
    engines = get_infinity_engines()  # 3 个独立引擎
    max_workers = min(len(engines), len(chunks))
    with ThreadPoolExecutor(max_workers=max_workers) as pool:
        futures = {
            pool.submit(encode_chunk, spec, idx % max_workers): spec[0]
            for idx, spec in enumerate(chunk_specs)
        }
        for future in as_completed(futures):
            chunk_idx, indices, hashes, encoded = future.result(timeout=900)
            _process_and_cache(indices, hashes, encoded)
            _report_progress(chunk_idx)
```
- 多个 chunk 分配到不同引擎，轮询调度（`idx % max_workers`）。
- `future.result(timeout=900)` 防止引擎卡死导致线程永久阻塞。
- 任一引擎异常时，取消剩余任务，**自动回退到 sentence-transformers 顺序模式**完成剩余 chunk，保证任务不中断。

**Step 4：性能统计**
```python
stats = {
    "totalImages": 5000,
    "encodedImages": 4800,             # 实际推理数
    "cacheHits": 200,                  # 跨数据集缓存命中数
    "totalInferenceSeconds": 45.2,
    "averageInferenceMsPerImage": 9.4,
    "batchSize": 64,
    "chunkSize": 256,
    "chunks": 19,
    "device": "cuda",
}
```
- 统计数据写入 `BGE_IMAGE_INFERENCE_STATS` 全局变量，随 embedding 元信息返回前端。

**2.8 倍加速的来源：**
- 原始方案：单引擎顺序处理所有 chunk，GPU 利用率低。
- 优化后：3 个 Infinity 引擎并行处理 3 个 chunk，GPU 复用率和吞吐量大幅提升。
- 实际加速比约 2.8 倍，未达理论 3 倍是因为引擎调度和 I/O 开销。

### 1.6 PCA 降维可视化

**函数：** `project_to_2d()`

```python
from sklearn.decomposition import PCA
projected = PCA(n_components=2, random_state=42).fit_transform(embeddings)
normalized = normalize_points(projected)  # 映射到 [-8, 8] 范围
```
- 768 维嵌入 → PCA → 2 维坐标。`random_state=42` 保证结果可复现。
- `normalize_points()` 以原点为中心归一化到 [-8, 8]，保证不同数据集间的散点图视觉一致。

**降级方案（BGE 不可用时）：** `apply_feature_projection()` + `build_feature_vector()`
- 手工构建 13 维特征向量：宽高比 + 检测框数量/面积 + 框中心均值 + 8 维哈希特征。
- 再用 PCA 降到 2D。精度不如语义嵌入，但能反映几何和类别差异。

### 1.7 Milvus 向量存储

**文件：** `backend/milvus_store.py`，使用 Milvus Lite（本地文件模式，数据文件 `.milvus-data.db`）。

| 集合 | 用途 | 关键字段 |
|------|------|---------|
| `datasets` | 数据集元信息 | dataset_id(PK), name, description, category_counts(JSON), split_counts(JSON), semantic_schema_version |
| `images` | 图片完整信息 | id(PK), dataset_id, filename, detections(JSON), metadata(JSON), embedding(768d FLOAT_VECTOR), embedding_2d(2d) |
| `embedding_cache` | 跨数据集嵌入缓存 | file_hash(PK, SHA-256), embedding(768d FLOAT_VECTOR) |

- `images.embedding` 字段建立 **FLAT 索引**（L2 距离），支持 `search_similar()` 向量相似度检索。
- `embedding_cache` 跨数据集共享：同一图片（SHA-256 一致）在任意数据集中只推理一次。
- 清洗建议（cleaning JSON）因可能超过 Milvus VARCHAR 上限（65535），改为文件存储：
  `{store_root}/{dataset_id}/cleaning.json`。

## 二、数据清洗与语义标注模块

### 2.1 kNN 离群检测

**实现：** `backend/dataset_service.py` 的 `compute_cleaning()`，纯 numpy 预计算。

**算法逻辑：**

1. 将所有图片的 **768 维高维嵌入**（非 2D 投影）L2 归一化，计算相似度矩阵 `sim = M @ M.T`（点积即余弦相似度）。
2. 对每张图取 **k 个最近邻**（默认 k=6，排除自身），计算平均余弦距离：
   ```
   outlier_score = 1 - mean(top_k_cosine_similarities)
   ```
   距离越大越孤立。
3. 用**分位线**划定离群：取 `outlier_percentile`（默认 0.95）分位值作为阈值，
   分数超过该值的标记为离群（约 **5%**）。
4. 按分数降序排序，截断到 `max_outliers`（默认 0=自动缩放为 `max(50, n×15%)`）。
5. 结果预计算后随 payload 下发，前端从 Zustand store 读取，**不再在浏览器跑 O(n²)**。

**与旧实现的区别：**
| 维度 | 旧（client-side TypeScript） | 新（server-side numpy） |
|------|---------------------------|----------------------|
| 向量空间 | 2D PCA 投影 | 768d 高维嵌入 |
| 距离 | 欧氏距离 | 余弦距离 |
| 阈值 | mean + 1.5σ（固定，约 6-7%） | 分位线（可控 ~5%） |
| 执行位置 | 浏览器 O(n²) | 后端 numpy 预计算 |
| 复杂度 | O(n²)，5000 张 ~1250 万次开方 | O(n²) 矩阵乘法，后端 ~1 秒 |

**可视化：** 散点图上离群点显示**红色双环光晕**（外环半透明 + 内环实线）。

### 2.2 互最近邻近重复识别

**实现：** `compute_cleaning()` 的 duplicates 分支，同样基于 768d 余弦空间。

**算法逻辑（互最近邻 / Mutual Nearest-Neighbor）：**

1. 计算完整相似度矩阵后，将自相似度（对角线）置为 `-∞`。
2. 对每张图 `i`，找到它相似度最高的邻居 `nn[i] = argmax(sim[i])`。
3. 判定条件：`i` 与 `nn[i]` **互为最近邻**（`nn[nn[i]] == i`）**且**余弦相似度 ≥ `dup_threshold`（默认 **0.98**）。
4. 满足条件的图片加入重复集合（去重），按相似度降序排序，截断到 `max_pairs`（默认 0=自动缩放为 `max(50, n×30%)`）。
5. 结果报告的是**重复图片数量**而非重复对数。

**为什么用互最近邻而非全配对阈值？**

车辆数据集的嵌入空间极度稠密（实测 cos≥0.93 的配对超过 190 万对），全配对阈值法会造成数量爆炸且语义不可靠。互最近邻的两大优势：
- **数量可控**：每张图至多有一个最近邻候选，重复图片数上界为 n。
- **语义更可靠**："你是我最像的、我也是你最像的"比"我俩距离小于某阈值"更严格，误报更低。

**与旧实现的区别：**
| 维度 | 旧（client-side） | 新（server-side） |
|------|-----------------|-----------------|
| 向量空间 | 2D PCA 投影 | 768d 高维嵌入 |
| 距离 | 欧氏距离 < 0.13 | 余弦相似度 ≥ 0.98 |
| 判定方式 | 全配对阈值 | 互最近邻 |
| 输出 | 重复对数（165573 对 / 5000 张） | 重复图片数（约几十张） |

**可视化：** 散点图上重复图片显示**蓝色双环光晕**（每张图独立标记，不再画配对连线）。

### 2.3 语义属性零样本自动标注

**6 个语义维度：**

| 维度 | 可选值 | 描述 |
|------|--------|------|
| `lighting` | bright, moderate, dim | 光照条件 |
| `viewpoint` | front, side, rear, overhead | 拍摄视角 |
| `blur` | sharp, motion-blur, out-of-focus | 清晰度 |
| `weather` | clear, cloudy, rain, snow, fog | 天气条件 |
| `timeOfDay` | day, dusk, night | 拍摄时间 |
| `environment` | indoor, urban-street, construction-site, rural-field, aerial-scene | 环境类型 |

#### 方案 A：BGE 零样本分类（默认）

利用 BGE-VL-Large 的**图像-文本联合嵌入空间**，通过余弦相似度实现零样本分类。

**Step 1：预编码文本提示**
```python
SEMANTIC_PROMPTS = {
    "lighting": {
        "bright": "a bright well lit image with strong overall illumination...",
        "moderate": "moderately lit image with balanced lighting...",
        "dim": "a dim or dark image with poor visibility...",
    },
    # ... 每个维度每个标签都有详细的英文视觉描述
}
```
- 所有文本提示通过 BGE 编码为 768d 向量，缓存为 `SEMANTIC_TEXT_EMBEDDINGS` 单例。
- 文本编码只执行一次，后续所有图片共享。

**Step 2：逐图分类**
```python
for image, image_embedding in zip(images, image_embeddings):
    for key, candidates in SEMANTIC_PROMPTS.items():
        scores = [
            cosine_similarity(image_embedding, text_embedding[label])
            for label, text_embedding in candidates
        ]
        best_label = argmax(scores)
        confidence = softmax(scores)[best_label]
        image["metadata"]["semantics"][key] = best_label
```

**为什么用零样本而非微调？** 不需要额外标注数据，新增维度只需添加文本描述。

#### 方案 B：GPT Vision 分类

当 `SEMANTIC_PROVIDER=gpt-vision` 时启用，适合需要更高精度的场景。图片缩放到 max 768px 后转 base64，调用 OpenAI 兼容 Chat API，返回 JSON 格式的语义标签。GPT 调用失败时自动降级到 BGE 零样本。

### 2.4 占位语义（Placeholder）

BGE 推理需要时间，但前端需要立即显示语义过滤选项。`derive_placeholder_semantics()` 基于文件名 + 主类别 + 分割的 SHA-256 哈希生成确定性伪随机语义标签。同一张图每次生成的占位值一致，BGE 推理完成后真实语义会替换占位值。

### 2.5 缓存验证

`is_semantic_cache_valid()` 在加载缓存数据集时校验三件事：
1. 语义 schema 版本是否匹配（`semantic-labels-2026-06-22-3`）
2. 语义 provider 是否匹配（bge / gpt-vision）
3. 所有图片的语义标签值是否在允许范围内

任一项不匹配则丢弃缓存，重新处理。

## 三、前端交互设计

### 3.1 核心视图

| 视图 | 组件 | 功能 |
|------|------|------|
| 图片网格 | `grid-view.tsx` | 响应式图片网格，SVG 归一化边界框叠加，懒加载（每批 200 张） |
| 散点图 | `scatter-view.tsx` | Canvas 2D 嵌入散点图，矩形/多边形/套索选择，颜色映射，清洗叠加层（离群红色光晕 + 重复蓝色光晕） |

### 3.2 散点图交互

**选择工具：**
- **矩形选择**：拖拽画矩形，框内点全部选中。
- **多边形选择**：点击添加顶点，双击闭合，射线法判断点是否在多边形内。
- **套索选择**：自由绘制路径，闭合后选中路径内所有点。

**颜色映射模式：** split（数据分割）、lighting（光照）、viewpoint（视角）、blur（清晰度）、weather（天气）、timeOfDay（时间）、environment（环境）。每种模式有独立的配色方案。

### 3.3 筛选系统

**Store：** `src/lib/store.ts`（Zustand），`getFilteredImages()` 按以下顺序逐级过滤：
1. 选中的 split（train/val/test）
2. 选中的目标类别（至少一个检测框匹配）
3. 散点图选区
4. 语义属性（维度内 OR 逻辑）
5. 搜索关键词（匹配文件名）

### 3.4 清洗建议展示

- 散点图工具栏的"清洗建议"按钮触发 `showCleaning` 状态。
- 数据来自后端预计算的 `payload.cleaning`，存入 Zustand store，`useCleaningIssues` hook 直接读取。
- 计数器显示「离群 N」「重复图片 M」。
- 悬浮提示：离群点显示"疑似离群点"，重复图片显示"疑似重复样本"。

## 四、配置参数速查

### BGE 推理

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| `BGE_VL_MODEL_PATH` | `E:\zzq\model\BGE-VL-large` | 模型权重路径 |

| `BGE_DEVICE` | auto | `cuda` 或 `cpu` |
| `BGE_USE_INFINITY` | `True` | 是否启用 Infinity 引擎 |
| `BGE_INFINITY_CONCURRENCY` | `3` | 引擎池大小（并行 chunk 数） |
| `BGE_EMBEDDING_CHUNK_SIZE` | `256` | 分块大小（每块图片数） |
| `BGE_PRELOAD_ON_STARTUP` | `True` | 启动时预加载模型 |

### 清洗建议

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| `CLEANING_DUP_THRESHOLD` | `0.98` | 重复判定余弦相似度下限（互最近邻） |
| `CLEANING_OUTLIER_PERCENTILE` | `0.95` | 离群分位线（约 5%） |
| `CLEANING_K_NEIGHBORS` | `6` | 离群计算 k 近邻数 |
| `CLEANING_MAX_PAIRS` | `0` | 重复图片数上限（0=自动缩放 `max(50, n×30%)`） |
| `CLEANING_MAX_OUTLIERS` | `0` | 离群数上限（0=自动缩放 `max(50, n×15%)`） |

## 五、关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 嵌入模型 | BGE-VL-Large（768d） | 图像-文本联合空间，支持零样本语义分类 |
| 推理引擎 | Infinity Embedding × 3 | 异步批处理 + 多引擎并行，2.8 倍加速 |
| 向量存储 | Milvus Lite（本地文件） | 零部署，支持 FLAT 向量索引和相似度搜索 |
| 缓存策略 | SHA-256 内容哈希 | 跨数据集共享，同一图片只推理一次 |
| 降维方法 | PCA（random_state=42） | 简单高效，结果可复现 |
| 离群检测 | kNN 平均余弦距离 + 分位线 | 高维语义空间，阈值可控（~5%） |
| 重复检测 | 互最近邻 + 余弦相似度 ≥ 0.98 | 数量可控，语义可靠，不因稠密爆炸 |
| 语义分类 | BGE 零样本（默认），GPT 降级 | 无需标注，新增维度零成本 |
| 清洗预计算 | 后端 numpy，随 payload 下发 | 前端秒出，不跑 O(n²) |
| 前端状态 | Zustand | 轻量、TypeScript 友好、selector 性能好 |
| ZIP 备份 | move 替代 copy | 省一次完整文件 I/O |
| 图片解析 | ThreadPoolExecutor 并行 | I/O 密集型，显著缩短解析时间 |
