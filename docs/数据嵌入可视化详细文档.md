# AI 数据分析平台 — 项目详细文档

## 项目总览

这是一个**计算机视觉数据集可视化分析平台**，全栈架构为 **Next.js 16 + React 19 前端 + Python FastAPI 后端**。核心功能是：用户上传一个 YOLO/Pascal VOC 格式的 ZIP 数据集，平台自动解析标注、提取图像语义嵌入、生成二维散点可视化，并提供数据清洗信号和语义属性自动标注。

整体分为两大模块：
1. **嵌入可视化分析模块** — 从 ZIP 解析到 BGE-VL-Large 嵌入提取、分块并行加速、PCA 降维可视化
2. **数据清洗与语义标注模块** — kNN 离群检测、近重复识别、多模态零样本语义分类

---

## 一、嵌入可视化分析模块

### 1.1 整体处理流水线

上传一个 ZIP 数据集后，后端按以下顺序处理：

```
ZIP 上传 → SHA-256 去重 → 解压 → 自动发现数据集根目录
→ 解析图片+标注(Pascal VOC / YOLO) → BGE-VL-Large 图像嵌入
→ 嵌入缓存(Milvus) → PCA 降维到 2D → 语义属性分类 → 返回完整 payload
```

关键设计：**两阶段返回策略**。
- 第一阶段：先返回基础数据（图片元信息、标注、类别统计），前端立即展示图片网格。
- 第二阶段：后台异步生成 BGE embedding，完成后更新散点图坐标和语义属性。

这样用户上传后不必等待模型推理，几秒内就能看到图片和标注。

### 1.2 ZIP 解压与数据集根目录自动发现

**入口函数：** `DatasetService.create_dataset_from_zip()`

**处理流程：**
1. 计算 ZIP 文件内容的 SHA-256 哈希，如果之前处理过完全相同的 ZIP，直接返回缓存结果（跳过全部解析和推理）。
2. 将 ZIP 解压到 `.dataset-store/{dataset_id}/extracted/` 目录。
3. 调用 `find_dataset_root()` 自动发现数据集根目录。

**根目录发现逻辑 (`find_dataset_root()`)：**
- 检查当前目录或一级子目录是否包含 `images/` 文件夹或 `train/val/test` 分割目录。
- 支持两种数据集布局：
  - **分割式布局**：`train/images/`, `val/images/`, `test/images/`
  - **扁平式布局**：`images/` + `labels/` + `xml/`（默认归入 train 分割）

**安全措施：** `_safe_extract_zip()` 会检查每个 ZIP 成员的路径，防止路径穿越攻击。

### 1.3 标注解析

支持两种主流标注格式：

**Pascal VOC XML (`parse_pascal_voc()`)：**
- 解析 XML 中的 `<object>` 元素，提取 `<name>`（类别）、`<bndbox>`（边界框）。
- 坐标归一化：将像素坐标 `(xmin, ymin, xmax, ymax)` 转为 `[x, y, w, h]` 的 0-1 归一化格式。
- 额外提取 `difficult` 和 `truncated` 属性。

**YOLO TXT (`parse_yolo_label()`)：**
- 每行格式：`class_id cx cy w h [confidence]`
- `cx, cy, w, h` 已经是归一化坐标，直接使用。
- 如果没有 confidence 列，默认为 1.0。
- 类别名用 `class_{id}` 格式（因为 YOLO txt 不含类别名）。

### 1.4 BGE-VL-Large 图像嵌入提取

**模型：** BGE-VL-Large，768 维图像-文本联合嵌入空间。

**两种推理模式（通过 `BGE_USE_INFINITY` 环境变量控制）：**

#### 模式 A：Sentence-Transformers（`use_infinity=False`）
```python
from sentence_transformers import SentenceTransformer
model = SentenceTransformer(model_path, trust_remote_code=True)
encoded = model.encode(image_paths, batch_size=64)
```
- 直接加载模型，批量编码图片路径。
- 需要调用 `configure_bge_sentence_transformer()` 修复兼容性问题：新版 sentence-transformers 会尝试索引 `pooler_output` 键，但 BGE-VL 的 CLIP 模块直接返回嵌入张量，导致 `too many indices for tensor of dimension 2` 错误。解决方案是将 `method_output_name` 从 `"pooler_output"` 改为 `None`。

#### 模式 B：Infinity Embedding Engine（`use_infinity=True`，默认）
```python
from infinity_emb import AsyncEmbeddingEngine, EngineArgs
engine = AsyncEmbeddingEngine.from_args(EngineArgs(
    model_name_or_path=model_path,
    engine="torch",
    device="cuda",
    bettertransformer=True,
    batch_size=64,
))
```

**引擎池架构：**
- 创建 `BGE_INFINITY_CONCURRENCY`（默认 4）个独立的 Infinity 引擎实例。
- 每个引擎在独立的守护线程中运行自己的 asyncio 事件循环。
- 初始化过程：
  1. 启动临时线程，创建新的 asyncio 事件循环。
  2. 创建 `AsyncEmbeddingEngine` 并调用 `engine.astart()` 初始化。
  3. 启动守护线程运行 `loop.run_forever()`。
  4. 等待初始化线程完成，确保引擎就绪。
- 线程安全提交：通过 `asyncio.run_coroutine_threadsafe(coro, loop)` 将编码任务提交到引擎的事件循环。

**为什么用 Infinity 而不是直接用 sentence-transformers？**
- Infinity 支持异步批处理、动态批处理队列、更好的 GPU 利用率。
- 多引擎池可以真正并行处理多个 chunk，实现 2.8 倍推理加速。

### 1.5 分块并行处理与嵌入缓存

**核心函数：** `get_cached_or_encode_image_embeddings()`

这是实现加速的关键。处理流程：

**Step 1：逐图查缓存**
```python
for image in images:
    digest = sha256(file_content)  # 文件内容哈希
    cached_vector = milvus.get_cached_embedding(digest)
    if cached_vector:
        vectors.append(cached_vector)  # 缓存命中
    else:
        missing_images.append(image)   # 需要推理
```
- 缓存键是文件内容的 SHA-256 哈希，不是文件名。这意味着同一张图片无论路径如何变化，都只推理一次。
- 缓存存储在 Milvus Lite 的 `embedding_cache` 集合中，跨数据集共享。

**Step 2：分块处理**
```python
chunk_size = 256  # BGE_EMBEDDING_CHUNK_SIZE
chunks = ceil(len(missing_images) / chunk_size)
```
- 将未缓存的图片分成多个 chunk，每个最多 256 张。
- 分块的目的是避免一次性加载过多图片导致 OOM。

**Step 3：并行编码（Infinity 模式下）**
```python
if concurrency > 1 and len(chunks) > 1:
    with ThreadPoolExecutor(max_workers=min(engines, chunks)) as pool:
        futures = {
            pool.submit(encode_chunk, spec, idx % max_workers): spec[0]
            for idx, spec in enumerate(chunk_specs)
        }
        for future in as_completed(futures):
            chunk_index, indices, hashes, encoded = future.result()
            # 写入缓存并更新 vectors
```
- 每个 chunk 分配到一个引擎（轮询分配：`idx % max_workers`）。
- 使用 `ThreadPoolExecutor` + `as_completed` 实现真正的并行推理。
- 每个 chunk 推理完成后立即写入缓存。

**Step 4：记录性能统计**
```python
stats = {
    "totalImages": 1000,
    "encodedImages": 800,        # 实际推理的图片数
    "cacheHits": 200,            # 缓存命中的图片数
    "totalInferenceSeconds": 12.5,
    "averageInferenceMsPerImage": 15.6,
    "batchSize": 64,
    "chunkSize": 256,
    "chunks": 4,
    "device": "cuda",
}
```

**2.8 倍加速的来源：**
- 原始方案：单引擎顺序处理所有 chunk。
- 优化后：4 个 Infinity 引擎并行处理 4 个 chunk，理论上 4 倍加速，实际因调度开销约 2.8 倍。

### 1.6 PCA 降维可视化

**函数：** `project_to_2d()`

```python
from sklearn.decomposition import PCA
projected = PCA(n_components=2, random_state=42).fit_transform(embeddings)
normalized = normalize_points(projected)  # 归一化到 [-8, 8]
```

- 768 维嵌入向量通过 PCA 降到 2 维。
- `random_state=42` 确保结果可复现。
- `normalize_points()` 将坐标归一化到 [-8, 8] 范围，以原点为中心，保证散点图布局一致。

**降级方案：** 如果 sklearn 不可用或样本数 < 2，直接取前两维。

### 1.7 确定性特征投影（BGE 不可用时的降级方案）

**函数：** `apply_feature_projection()` + `build_feature_vector()`

当 BGE-VL-Large 模型不可用时（如模型路径不存在、推理失败），使用手工特征构建替代方案：

```python
feature_vector = [
    width / height,           # 宽高比
    len(detections),          # 检测框数量
    sum(bbox_w * bbox_h),     # 检测框总面积
    mean_center_x,            # 检测框中心 x 均值
    mean_center_y,            # 检测框中心 y 均值
    *hash(image_id + label),  # 8 维哈希特征（注入类别和 ID 信息）
]
```
- 共 13 维特征向量，再 PCA 降到 2 维。
- 虽然不如 BGE 语义嵌入精确，但能大致反映图片的几何和类别差异。

### 1.8 Milvus 向量存储

**文件：** `backend/milvus_store.py`

使用 **pymilvus**（Milvus Lite，本地文件模式），定义 3 个集合：

| 集合名 | 用途 | 关键字段 |
|--------|------|---------|
| `datasets` | 数据集元信息 | id, name, description, categoryCounts, splitCounts, semanticSchemaVersion |
| `images` | 每张图片的完整信息 | id, datasetId, filename, detections(JSON), metadata(JSON), embedding(768维向量), embedding2d(2维向量) |
| `embedding_cache` | 跨数据集嵌入缓存 | digest(SHA-256), vector(768维) |

- `images` 集合在 `embedding` 字段上建立 FLAT 索引（L2 距离），支持 `search_similar()` 向量相似度查询。
- 嵌入缓存是跨数据集共享的，同一张图片在不同数据集中只推理一次。

## 二、数据清洗与语义标注模块

### 2.1 kNN 离群检测

**文件：** `src/lib/cleaning/detectors.ts`
**函数：** `detectOutliers()`

**算法逻辑：**
1. 对每张图片，计算它在 2D 嵌入空间中与所有其他图片的欧氏距离。
2. 按距离排序，取最近的 k 个邻居（默认 k=6）。
3. 计算 k 个邻居距离的平均值，作为该图片的**离群分数**。
4. 计算所有图片离群分数的均值 μ 和标准差 σ。
5. 如果某图片的离群分数 > μ + 1.5σ，则标记为**疑似离群点**。

```typescript
// 伪代码
for each image:
    distances = [euclidean(image, other) for other in all_images]
    distances.sort()
    k_nearest = distances[1:k+1]  // 排除自身
    outlier_score = mean(k_nearest)

threshold = mean(all_scores) + 1.5 * std(all_scores)
outliers = [img for img in images if img.score > threshold]
```

**复杂度：** O(n²)（n 为图片数量），因为需要计算所有图片对之间的距离。

**可视化：** 在散点图上，离群点显示为红色光晕（双环：外环半透明 + 内环实线）。

### 2.2 欧氏距离近重复识别

**函数：** `detectDuplicates()`

**算法逻辑：**
1. 遍历所有图片对 (i, j)，计算 2D 嵌入空间中的欧氏距离。
2. 如果距离 < ε（默认 0.13），标记为**疑似重复样本**。
3. 按距离排序返回（最近的对排在前面）。

```typescript
// 伪代码
for i in range(n):
    for j in range(i+1, n):
        dist = euclidean(images[i].embedding2d, images[j].embedding2d)
        if dist < 0.13:
            duplicates.add((i, j, dist))
```

**ε 的选择：** 0.13 是基于 [-8, 8] 坐标范围的经验值。太小会漏检，太大会误报。

**可视化：** 在散点图上，重复对用蓝色半透明线段连接，端点有圆环标记。

### 2.3 语义属性自动标注

**6 个语义维度：**

| 维度 | 可选值 | 描述 |
|------|--------|------|
| `lighting` | bright, moderate, dim | 光照条件 |
| `viewpoint` | front, side, rear, overhead | 拍摄视角 |
| `blur` | sharp, motion-blur, out-of-focus | 清晰度 |
| `weather` | clear, cloudy, rain, snow, fog | 天气条件 |
| `timeOfDay` | day, dusk, night | 拍摄时间 |
| `environment` | indoor, urban-street, construction-site, rural-field, aerial-scene | 环境类型 |

**两种分类方案：**

#### 方案 A：BGE 零样本分类（默认）

**核心思路：** 利用 BGE-VL-Large 的图像-文本联合嵌入空间，通过余弦相似度实现零样本分类。

**Step 1：预编码文本提示**
```python
# 每个维度的每个标签都有详细的英文视觉描述
SEMANTIC_PROMPTS = {
    "lighting": {
        "bright": "a bright well lit image with strong overall illumination and clearly visible details",
        "moderate": "moderately lit image with balanced lighting",
        "dim": "a dim or dark image with poor visibility",
    },
    # ... 其他维度
}
```
- 将所有文本提示通过 BGE 编码为向量，缓存为 `SEMANTIC_TEXT_EMBEDDINGS` 单例。
- 文本编码只需要做一次，后续所有图片共享。

**Step 2：逐图分类**
```python
for image, image_embedding in zip(images, image_embeddings):
    for key, candidates in SEMANTIC_PROMPTS.items():
        scores = [cosine_similarity(image_embedding, text_embedding) for label, text_embedding in candidates]
        best_label = argmax(scores)
        confidence = softmax(scores, best_label)
        image["metadata"]["semantics"][key] = best_label
        image["metadata"]["semanticMeta"][key] = {"source": "bge-zero-shot", "confidence": confidence}
```

**余弦相似度计算：**
```python
def cosine_similarity(a, b):
    dot = sum(ai * bi for ai, bi in zip(a, b))
    norm_a = sqrt(sum(ai * ai for ai in a))
    norm_b = sqrt(sum(bi * bi for bi in b))
    return dot / (norm_a * norm_b)
```

**置信度计算（Softmax）：**
```python
def softmax_confidence(scores, best_index):
    max_score = max(scores)
    exps = [exp(s - max_score) for s in scores]
    return exps[best_index] / sum(exps)
```

**为什么用零样本而不是微调分类器？**
- 不需要额外标注数据，直接利用预训练模型的语义理解能力。
- 新增语义维度只需添加文本描述，不需要重新训练。
- 对于"光照"、"天气"这类通用视觉属性，预训练模型已经有很好的理解。

#### 方案 B：GPT Vision 分类

**适用场景：** 需要更高精度的语义理解，或 BGE 零样本效果不理想的维度。

**流程：**
1. 将图片转为 base64 JPEG（缩放到最大 768px）。
2. 调用 OpenAI 兼容的 Chat API，附带系统提示和语义选项 JSON。
3. 解析返回的 JSON，验证所有标签是否在允许范围内。

```python
def classify_image_semantics_with_gpt(image, config):
    image_url = image_to_data_url(image_path, max_size=768)
    response = call_openai_compatible_chat(
        base_url=config["baseUrl"],
        api_key=config["apiKey"],
        model=config["model"],  # 默认 gpt-5.5
        image_url=image_url,
    )
    parsed = parse_semantic_json(response)
    return normalize_semantic_response(parsed)
```

**GPT Prompt 设计：**
```
Analyze the image and choose exactly one label id for each semantic field.
Use only the label ids listed below. Each label has an English visual description;
choose the closest visible match. Do not invent labels.
Semantic options: {options_json}
Return JSON only, with no markdown, in this exact shape:
{"lighting":"","viewpoint":"","blur":"","weather":"","timeOfDay":"","environment":""}
```

**降级机制：** 如果 GPT Vision 调用失败（网络错误、API 限流等），自动降级到 BGE 零样本分类。

### 2.4 占位语义（Placeholder Semantics）

**问题：** BGE 推理需要时间，但前端需要立即显示语义属性。

**解决方案：** `derive_placeholder_semantics()`

```python
def derive_placeholder_semantics(stem, primary_label, split):
    units = hash_units(f"{stem}:{primary_label}:{split}", 8)
    return {
        "lighting": pick_option("lighting", units[0]),
        "viewpoint": pick_option("viewpoint", units[1]),
        # ...
    }
```

- 基于图片文件名 + 主类别 + 分割 的 SHA-256 哈希，生成确定性的伪随机语义标签。
- 同一张图片每次生成的占位语义是一致的。
- 当 BGE 推理完成后，真实语义会替换这些占位值。

### 2.5 缓存验证

**函数：** `is_semantic_cache_valid()`

当加载缓存的数据集时，需要验证语义缓存是否仍然有效：

```python
def is_semantic_cache_valid(payload):
    # 1. 检查语义 schema 版本是否匹配
    if source["semanticSchemaVersion"] != current_version:
        return False
    # 2. 检查语义 provider 是否匹配
    if source["semanticProvider"] != current_provider:
        return False
    # 3. 检查所有图片的语义值是否在允许范围内
    for image in images:
        for key, allowed_values in SEMANTIC_OPTIONS.items():
            if image["metadata"]["semantics"][key] not in allowed_values:
                return False
    return True
```

- 如果 schema 版本、provider 或标签范围发生变化，缓存会被丢弃并重新处理。
- 这保证了语义标签的一致性。

## 四、关键技术决策与权衡

| 决策 | 选择 | 理由 |
|------|------|------|
| 嵌入模型 | BGE-VL-Large | 768 维图像-文本联合空间，支持零样本分类 |
| 推理引擎 | Infinity Embedding | 异步批处理、多引擎并行、GPU 利用率高 |
| 向量存储 | Milvus Lite | 本地文件模式，无需部署服务，支持向量索引 |
| 缓存策略 | SHA-256 内容哈希 | 跨数据集共享，同一图片只推理一次 |
| 降维方法 | PCA | 简单高效，随机种子保证可复现 |
| 离群检测 | kNN 平均距离 | 直观，可调节 k 和阈值 |
| 重复检测 | 欧氏距离阈值 | 简单有效，阈值可调 |
